Implementation Plan: shadcn/ui Adoption, Component Library Integration, Demo Pages, and Accent Theming

Date: 2026-03-18
Project: web-vite-react-fastapi-postgres-shadcn
Scope: Planning only. No implementation in this phase.

1. Objective

Adopt the current official shadcn/ui component catalog into the frontend codebase in a way that:

- fits the existing Vite + React + TypeScript structure,
- preserves the repo's feature-oriented organization,
- provides a reusable shared component layer under the app's code structure,
- includes sample/demo pages that prove the components work,
- introduces a controllable theme system where accent colors can be changed safely,
- avoids a one-off UI dump that becomes hard to maintain.


2. What The Repo Looks Like Today

Current frontend state observed in this repository:

- Frontend root: frontend/
- App runtime: React 19 + Vite + TypeScript
- Routing: react-router-dom with a single main route in frontend/src/app/router.tsx
- Shared UI primitives currently present:
  - frontend/src/components/ui/button.tsx
  - frontend/src/components/ui/input.tsx
  - frontend/src/components/ui/textarea.tsx
- Styling model: hand-authored global CSS in frontend/src/styles/globals.css
- No Tailwind setup present
- No shadcn CLI setup present
- No components.json present
- No @ alias configured in TypeScript/Vite in the shadcn convention
- No theme provider or runtime accent switcher present

Implication:

The app is not currently on the normal shadcn foundation. A clean implementation will require a design-system foundation phase first, then component ingestion, then demo composition.

4. Recommended Implementation Direction

Recommended approach:

- Use official shadcn CLI and registry patterns rather than recreating components manually.
- Move the frontend onto the expected Tailwind + CSS variables foundation.
- Keep shared design-system code in frontend/src/components/ui/.
- Keep app-specific demo compositions in frontend/src/features/showcase/.
- Keep business pages like products separate so the design system stays reusable.

Do not do this:

- Do not hand-port 50+ components from docs by copying snippets one by one.
- Do not keep mixing old handcrafted button/input styling with new shadcn primitives.
- Do not put every demo and every shared primitive directly into the products feature.
- Do not hardcode accent color choices inside individual components.


5. Foundation Changes Required Before Any Component Import

Phase 1 must establish the base expected by shadcn/ui.

5.1 Tooling and dependencies

Add frontend dependencies typically required by shadcn/ui and its ecosystem:

- tailwindcss
- @tailwindcss/vite or the project-approved Tailwind Vite integration
- class-variance-authority
- clsx
- tailwind-merge
- lucide-react
- next-themes equivalent is not needed in Vite specifically, but a theme manager pattern may still be useful
- component-specific dependencies introduced by shadcn CLI:
  - Radix packages
  - embla-carousel-react
  - sonner
  - recharts
  - react-day-picker
  - input-otp
  - tanstack/react-table
  - vaul or drawer dependency if required by the current registry

Implementation note:

Dependencies should be added incrementally through the CLI as components are installed, but the plan should anticipate them up front to avoid dependency surprise later.

5.2 Alias and config alignment

Update frontend configuration to align with shadcn conventions:

- Add @/* path alias in frontend/tsconfig.app.json
- Add matching resolve alias in frontend/vite.config.ts
- Ensure frontend/src/lib/utils.ts exposes the canonical cn() helper

5.3 shadcn project initialization

Initialize shadcn for Vite in frontend/ with:

- style: new-york
- css variables: true
- base color: neutral or slate
- icon library: lucide
- aliases:
  - components -> @/components
  - ui -> @/components/ui
  - lib -> @/lib
  - hooks -> @/hooks

Recommendation:

Use new-york + CSS variables + neutral base color.

Reason:

- It is the most flexible starting point.
- It works well with runtime accent overrides.
- It avoids overcommitting to a branded palette too early.
- It pairs better with a "themeable application shell" than a heavily opinionated base.

5.4 Global CSS migration

Replace the current handcrafted global styling foundation with a layered system:

- Tailwind base/utilities
- shadcn variable tokens
- a small app-shell layer for brand-specific layout styling

Plan for frontend/src/styles/globals.css:

- preserve only truly global layout rules,
- define :root and .dark token sets,
- map accent and semantic tokens,
- avoid retaining old .button/.input/.textarea class contracts after the migration,
- keep the app's visual identity in layout-level classes rather than shared primitives.

7. Theme Strategy

The theme system is the most important architectural decision because the user explicitly wants accent color control.

7.1 Recommended theming model

Use CSS variable theming, which the official docs recommend, instead of utility-class-only theming.

Base tokens to own:

- --background
- --foreground
- --card
- --card-foreground
- --popover
- --popover-foreground
- --primary
- --primary-foreground
- --secondary
- --secondary-foreground
- --muted
- --muted-foreground
- --accent
- --accent-foreground
- --destructive
- --destructive-foreground
- --border
- --input
- --ring
- --chart-1 through --chart-5
- sidebar token family if Sidebar is included

7.2 Accent color control

Recommended implementation:

- Keep the base neutral theme stable.
- Allow runtime switching of an accent preset.
- Accent preset should update:
  - --primary
  - --primary-foreground
  - --accent
  - --accent-foreground
  - --ring
  - optionally chart tokens for visual coherence

Suggested preset families:

- Amber
- Blue
- Emerald
- Rose
- Violet
- Zinc-neutral fallback

7.3 Best practical theme choice for this repo

Recommendation:

- Light and dark mode support
- Neutral base surfaces
- Accent-driven branding
- New York style components

Why this is the best fit:

- The current app already has a strong branded surface style, but it is hardcoded.
- Neutral surfaces with configurable accent tokens will let the app keep personality while making the system reusable.
- Accent switching is more maintainable than trying to dynamically regenerate entire theme palettes.

7.4 Persistence and API surface

Theme state should be:

- stored in localStorage,
- applied at app boot to avoid flash,
- exposed through a ThemeProvider/context,
- controlled by a small ThemePanel component in the sample pages.

11. Functional Expectations For The Demo Pages

The demo pages should not be static screenshots. They need working interactions.

Required functional behaviors:

- dialogs open and close correctly,
- dropdowns, popovers, and menus manage focus correctly,
- theme toggles and accent changes apply live,
- data table supports sorting/filtering at minimum,
- chart renders with theme tokens,
- calendar/date picker can select values,
- form examples show validation or state transitions,
- notifications render through the global provider,
- sidebar collapses or switches responsively if included.

14. Accessibility and Quality Requirements

The implementation should explicitly preserve shadcn/Radix accessibility behavior.

Checklist:

- keyboard navigation works for overlays and menus,
- focus rings use tokenized ring colors,
- interactive states remain visible in both light and dark mode,
- contrast remains acceptable across accent presets,
- reduced-motion-safe behavior should not be broken by custom transitions,
- demo pages remain usable on mobile widths.

16. Risks and Constraints

16.1 "All components" is dependency-heavy

Adding the full current catalog will increase package count and frontend code size. This is manageable, but only if installed through a phased, dependency-aware process.

16.2 Current CSS will conflict with shadcn primitives

The current globals define .button, .input, and other element assumptions. During migration these will create styling conflicts unless retired carefully.

16.3 Data table and chart are not trivial

These components are compositions, not tiny primitives. They require additional patterns, sample data, and testing.

16.4 Sidebar requires layout decisions

If included, the app shell must define whether showcase routes and business routes share the same sidebar model or use separate shells.

16.5 Theme flexibility can become inconsistent

If too many tokens are made runtime-editable, the system becomes fragile. Only accent-related tokens should be dynamic at first.


17. Recommended Execution Order

This is the order I recommend for the actual implementation phase:

1. Set up Tailwind, aliases, cn helper, and shadcn initialization in frontend/
2. Replace current button/input/textarea foundation with shadcn primitives
3. Introduce theme tokens, ThemeProvider, dark mode, and accent preset switching
4. Migrate the products feature to the new shared UI
5. Add the showcase route tree and category pages
6. Import the remaining official component catalog in dependency-aware batches
7. Wire global toast/sonner providers
8. Validate builds, routes, theming, and responsive behavior


18. Definition Of Done

The work should be considered complete only when:

- the frontend is on a proper shadcn-compatible foundation,
- the official current shadcn core component catalog has been integrated into the codebase,
- the products feature uses the new shared UI where appropriate,
- showcase pages exist and demonstrate the major component categories with working interactions,
- accent theming can be changed at runtime through a controlled theme panel,
- light/dark mode works,
- the app builds successfully and the demo pages are functional.

20. Planning Conclusion

This should be implemented as a design-system migration with a showcase feature, not as a one-time paste of shadcn examples.

The correct technical direction is:

- shadcn CLI-based component integration,
- Tailwind + CSS variable theming,
- neutral base + runtime accent presets,
- shared primitives under components/ui,
- demo routes under a dedicated showcase feature,
- selective migration of the existing products feature onto the new system.

That path will satisfy the user's request while keeping the repo maintainable.
